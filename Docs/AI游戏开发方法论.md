# AI 时代的游戏开发方法论

> 核心理念：不是 AI 替你做了游戏，而是你学会了用 AI 能理解的方式去创造

---

## 📖 为什么用 AI 做游戏？

### 三个动机

1. **复杂度验证**
   - 游戏是个人项目里综合难度最高的品类
   - 策划、美术、音效、开发、平衡性缺一不可
   - 如果 AI 能帮我一个人闭环这些，以后其他简单的事大概率手到擒来

2. **热爱驱动**
   - 亲手做一款属于自己的游戏
   - 把脑子里"更好的版本"变成现实
   - 创作本身比消费更有意思

3. **抓住灵感**
   - Steam 上有个叫「Bad North」的游戏
   - 画风唯美、操作有趣，但兵种和技能太少
   - 能不能用 AI 把"更丰富的版本"做出来？

### 核心洞察

> **AI 不会替代创作者，但它正在让"一个人 = 一个团队"变成现实**

## 🎯 AI 游戏开发的三个阶段

### 阶段 1：摸索期 - Vibe Coding（氛围编程）

**目标**：快速做出游戏原型

**特点**：
- 不关心代码质量
- 只求能跑起来
- 快速验证玩法
- 积累问题清单

**做法**：
```
你知道 Bad North 这个游戏吗？
生成一个和 Bad North 非常接近的游戏，
3D，角色可以实时移动和战斗，岛屿防御玩法，简洁策略操作。
修正代码以修复以上错误...
```

**结果**：
- ✅ 快速得到可运行的 Demo
- ❌ 代码结构混乱
- ❌ Bug 多得改不完
- ❌ 无法继续迭代

**教训**：
> Vibe Coding 能帮你快速验证想法，但无法支撑长期开发

---

### 阶段 2：重构期 - Spec Coding（按规范编程）

**问题爆发**：
- 代码结构混乱，bug 多得改不完
- 单文件巨大，模型上下文吃不下
- 没有记忆，每次对话从零开始
- 无法自动验证，只能靠人肉操作
- 性能瓶颈，移动端帧率低且不稳定

**关键认知**：
> AI 的能力已经足够好了，但如果项目本身的结构不是为它设计的，能力再强也发挥不出来

**核心原则**：
1. **为 AI 设计代码结构** - 让 AI 能理解项目的所有实现
2. **All in One** - 游戏项目的一切都由代码直接生成

**什么是 "为 AI 设计" 的代码结构？**

传统项目：
```
数值配置：Excel 表格
美术资源：Blender/PS 工程文件
音效资源：音频工程文件
游戏逻辑：代码
→ AI 只能看到代码部分，其他都是"黑箱"
```

AI-Native 项目：
```
数值配置：代码里的配置对象
美术资源：代码生成的几何体和材质
音效资源：代码实时合成
游戏逻辑：代码
→ AI 能读懂每一行，改动每一处
```

---

### 阶段 3：成熟期 - AI 协作范式

**"言出法随"的开发体验**

传统开发需要：
- 策划改表 → 程序实现 → 美术出资源 → 音频师改音 → TA 调效果
- 4-5 个工种、3-4 种工具、N 天时间

AI 协作只需要：
- 读代码，改代码
- 一种操作、一个对话、几秒钟

**真实案例**：

```
我：看一下弓箭手的攻击流程，把箭矢飞行速度调快一点

AI：我可以同时看到：
- 数值配置（GameConfig.js）
- 战斗逻辑（UnitCombat.js）
- 视觉表现（CombatEffects.js）
- 音效触发（SynthSFX.js）

调整后，速度变快了，音效也自动适配了新的飞行时间
```

## 🏗️ AI-Native 项目的核心选择

### 选择 1：技术栈

**选项 A：使用 Unity/Cocos/UE**
```
✅ 优点：
- 成熟引擎，功能完整
- 大量教程和资源
- 性能优化好

❌ 缺点：
- 不是为 AI 设计的
- 需要编辑器操作
- 资源文件 AI 看不到
- 学习曲线陡峭
```

**选项 B：All-in-Code（Web 技术栈）**
```
✅ 优点：
- AI 完全透明
- 迭代速度极快
- 不需要额外工具
- 跨平台

❌ 缺点：
- 需要自己造"引擎"
- 性能需要自己优化
- 视觉效果有限制
```

**建议**：
- 快速验证玩法 → 用 Unity
- 长期 AI 协作 → All-in-Code

---

### 选择 2：架构设计

**核心原则**：**为 AI 设计的代码结构**

以 Unity 为例，即使使用引擎，也可以做 AI-Native 设计：

```csharp
// ❌ 传统做法：数据在 Inspector 里
public class Unit : MonoBehaviour {
    public float maxHealth = 100;  // AI 看不到
    public float moveSpeed = 5;    // AI 看不到
}

// ✅ AI-Native 做法：数据在代码里
public class UnitConfig {
    public static readonly UnitConfig ARCHER = new UnitConfig {
        maxHealth = 80,
        moveSpeed = 6,
        attackRange = 10,
        attackDamage = 15
    };
}

public class Unit : MonoBehaviour {
    public UnitConfig config;  // AI 能看到所有配置
}
```

**五层架构示例**：

```
┌─────────────────────────────────────────────┐
│  入口层 Entry                                │
│  main.js — 启动 · 环境检测 · 平台适配          │
├─────────────────────────────────────────────┤
│  核心层 Core                                 │
│  EventBus · ServiceLocator · Events          │
├─────────────────────────────────────────────┤
│  管理器层 Managers                           │
│  GameManager · Camera · Input · Audio        │
├─────────────────────────────────────────────┤
│  系统层 Systems                              │
│  AI · Combat · Wave · Economy · Effects      │
├─────────────────────────────────────────────┤
│  游戏对象层 GameObjects                      │
│  World · Squad · Unit · Enemy                │
├─────────────────────────────────────────────┤
│  基础层 Foundation                           │
│  Factories · Utils · Materials · Config      │
└─────────────────────────────────────────────┘
```

## 🎨 AI-Native 的三大系统

### 1. 地图系统：用数学"长"出来的岛屿

**不用美术，用算法**

- 柏林噪声生成高度图
- 强制沙滩边界
- 地形平滑处理
- 路径连通性验证
- 自动检查陆地面积

**优势**：
- 无穷无尽的地图
- AI 可以调整任何参数
- 每张地图都经过自动验证

### 2. 角色系统：代码拼出来的小人

**Lowpoly 风格 = "粗糙"变"风格"**

- 圆柱体 = 身体
- 球体 = 头
- 长方体 = 手臂和腿
- 加上描边后处理 = 三渲二手绘风

**优势**：
- 不需要 3D 模型文件
- AI 可以调整任何细节
- 代码即资产

### 3. 音频系统：零音频文件的实时合成

**所有声音用 Web Audio API 实时合成**

- 背景音乐：正弦波合成
- 攻击音效：噪声滤波
- 脚步声：随机触发短音

**最大优势**：
- 声音能和游戏状态深度联动
- 子弹时间 → 音效真的"慢放"
- AI 知道技能时长 → 音效自动适配

## 🤝 AI 协作的核心技巧

### 1. 用对 AI 模型

- **Claude**：写逻辑（代码结构、系统设计）
- **Gemini**：写视觉（审美、颜色、动画）
- **GPT**：Debug（定位问题、分析错误）

### 2. 给 AI 完整上下文

**❌ 不要这样**：
```
帮我优化这个函数
[粘贴 50 行代码]
```

**✅ 应该这样**：
```
这是游戏的弓箭手攻击流程：
1. 配置：GameConfig.js 第 45-60 行
2. 逻辑：UnitCombat.js 第 120-180 行
3. 视觉：CombatEffects.js 第 80-100 行
4. 音效：SynthSFX.js 第 200-220 行

问题：箭矢飞行速度太慢
目标：提高到 1.5 倍，并自动调整音效时长
```

### 3. 让 AI 自己动手

**❌ 不要**：
```
你告诉我怎么改，我自己来
```

**✅ 要**：
```
直接修改这 4 个文件，确保所有系统协调
```

### 4. 建立记忆系统

- 用 `@README.md` 让 AI 记住项目结构
- 用 `@CHANGELOG.md` 让 AI 知道最近改动
- 用 `@TODO.md` 让 AI 知道待办事项

## 📊 成功的关键：妥协的艺术

### 传统游戏开发的妥协
- 想要更好的画面 → 需要更好的美术
- 想要更好的音效 → 需要录音棚
- 想要更复杂的玩法 → 需要更大的团队

### AI 时代游戏的妥协
- **放弃精美画面** → Lowpoly 风格
- **放弃录音棚音效** → 实时音频合成
- **放弃商业引擎便利** → All-in-Code

### 换来的是什么？
- **整个游戏对 AI 完全透明**
- **美术、音频、逻辑、配置、特效，全在同一个代码仓库里**
- **AI 能读懂每一行，改动每一处**

## 🎯 总结：三个核心观点

### 1. AI 不会替代创作者
AI 是工具，你是创作者。AI 帮你突破技术门槛，但创意和品味还是你自己。

### 2. 游戏是妥协的艺术
和 AI 一起做游戏，核心不是 AI 多强，而是你愿意在正确的方向上做出什么妥协和让步。

### 3. 不是 AI 替你做了游戏，而是你学会了用 AI 能理解的方式去创造
这是最重要的洞察：改变你的开发方式，让 AI 成为你的"超级助手"。
